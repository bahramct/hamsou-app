# دستورالعمل کار ایجنت توسعه‌دهنده (Agent Guidelines)

## مقدمه
این مستندات مجموعه کاملی از قوانین، دستورالعمل‌ها و رویه‌های کاری است که ایجنت توسعه‌دهنده Hamsou باید از آن پیروی کند. هدف این است که هر کسی سورس کد را مشاهده کند، متوجه شود چگونه پروژه توسعه یافته است و چه قوانینی در پیاده‌سازی رعایت شده است.

---

## ۱. تخصص‌ها و قابلیت‌ها

### ۱.۱. تخصص‌های فنی
- **Next.js 16 با App Router**: تسلط کامل بر سیستم مسیریابی مدرن Next.js
- **TypeScript 5**: برنامه‌نویسی با قواعد سخت‌گیرانه و type-safe
- **Tailwind CSS 4**: طراحی ریسپانسیو و مدرن
- **Prisma ORM**: کار با دیتابیس SQLite با استفاده از Prisma
- **shadcn/ui**: استفاده از کامپوننت‌های آماده و استاندارد
- **RESTful API**: طراحی و پیاده‌سازی API endpoints
- **State Management**: Zustand برای client state و TanStack Query برای server state
- **Real-time Communication**: WebSocket و Socket.io برای ارتباطات بلادرنگ
- **AI Integration**: استفاده از z-ai-web-dev-sdk برای قابلیت‌های هوش مصنوعی

### ۱.۲. قابلیت‌های AI Skills
- **LLM**: ایجاد چت‌بات و مکالمه هوشمند
- **VLM**: تحلیل و درک تصاویر
- **ASR**: تبدیل گفتار به متن
- **TTS**: تبدیل متن به گفتار
- **Image Generation**: تولید تصویر با AI
- **Video Generation**: تولید ویدیو با AI
- **Web Search**: جستجو در وب برای اطلاعات به‌روز
- **Web Reader**: استخراج محتوا از صفحات وب

## ۲. نظم و ترتیب انجام کارها

### ۲.۱. ترتیب پیاده‌سازی یک فیچر
۱. **تحلیل درخواست**: درک کامل نیاز کاربر و مقصد نهایی
۲. **بررسی قوانین**: مرور دستورالعمل‌ها و محدودیت‌های پروژه
۳. **برنامه‌ریزی**: تعیین مراحل و ایجاد todo list با TodoWrite
۴. **پیاده‌سازی فرانت‌اند**: ابتدا UI را برای کاربر قابل مشاهده می‌کنیم
۵. **پیاده‌سازی بک‌اند**: سپس API و منطق سمت سرور را اضافه می‌کنیم
۶. **تست فانکشنالیتی**: بررسی کامل عملکرد فیچر
۷. **تست‌های Trace**: اطمینان از عدم خطا و عملکرد صحیح
۸. **تأیید کاربر**: منتظر تأیید نهایی از طرف کاربر
۹. **به‌روزرسانی مستندات**: ویرایش roadmap-progress پس از تأیید

### ۲.۲. ترتیب ایجاد فایل‌ها برای صفحات جدید
۱. ابتدا صفحه فرانت‌اند در `src/app/[page]/page.tsx`
۲. سپس API routes در `src/app/api/[endpoint]/route.ts`
۳. در صورت نیاز، کامپوننت‌های جداگانه در `src/components/`
۴. در صورت نیاز، تایپ‌ها در `src/types/`
۵. در صورت نیاز، ابزارها در `src/lib/`

## ۴. قوانین پیاده‌سازی فیچرها

### ۴.۱. توسعه فرانت‌اند
- استفاده از `'use client'` برای کامپوننت‌های کلاینت
- استفاده از کامپوننت‌های موجود در `src/components/ui/`
- طراحی ریسپانسیو با Mobile-First approach
- استفاده از متغیرهای Tailwind (bg-primary, text-primary-foreground, ...)
- عدم استفاده از رنگ‌های indigo یا blue مگر درخواست صریح
- پیاده‌سازی loading states و error handling
- استفاده از Framer Motion برای انیمیشن‌های ظریف

### ۴.۲. توسعه بک‌اند
- استفاده از `'use server'` در API routes
- احراز هویت با verifyToken در تمام APIهای محافظت شده
- استفاده از Prisma برای عملیات دیتابیس: `import { db } from '@/lib/db'`
- در صورت نیاز به تغییر schema:
  1. ویرایش `prisma/schema.prisma`
  2. اجرای `bun run db:push`
- پیاده‌سازی error handling مناسب
- استفاده از status codes صحیح HTTP

### ۴.۳. استایل و UI/UX
- **رنگ‌های مجاز**: فقط Tailwind built-in variables
- **رنگ‌های ممنوع**: indigo، blue (مگر درخواست صریح)
- **پدینگ کارت‌ها**: p-4 یا p-6 برای محتوا، gap-4 یا gap-6 برای فاصله
- **لیست‌های طولانی**: max-h-96 overflow-y-auto با custom scrollbar
- **فوتر استیکی**: min-h-screen flex flex-col + mt-auto برای footer
- **تم** : پشتیبانی از light/dark mode با next-themes

### ۴.۴. WebSocket و Real-time
- WebSocket باید در mini-service جداگانه با پورت اختصاصی باشد
- فایل در `mini-services/[service-name]/index.ts`
- کانکشن فرانت‌اند: `io("/?XTransformPort={Port}")` با path = "/"
- هرگز استفاده نکنید: `io("http://localhost:{Port}")`
- هر mini service باید `bun run dev` خود را داشته باشد با auto-restart

## ۵. نباید‌ها (Forbidden Practices)

### ۵.۱. محدودیت‌های فنی
❌ **تغییر Framework**: Next.js 16 و TypeScript 5 غیرقابل تغییر هستند
❌ **استفاده از کتابخانه‌های خارجی**: مگر با مجوز کاربر
❌ **نوشتن absolute path در API requests**: باید relative path باشد
❌ **استفاده از z-ai-web-dev-sdk در کلاینت**: فقط در بک‌اند مجاز است
❌ **نوشتن port در URL**: استفاده از `?XTransformPort={Port}`
❌ **استفاده از Server Actions**: باید API routes استفاده شود
❌ **اجرای `bun run build`**: فقط dev mode مجاز است
❌ **تغییر پورت dev server**: فقط پورت 3000 مجاز است

### ۵.۲. محدودیت‌های کدنویسی
❌ **عدم استفاده از TypeScript**: تمام کد باید type-safe باشد
❌ **عدم رعایت responsive design**: همه چیز باید موبایل-فرست باشد
❌ **ساخت مجدد کامپوننت‌های shadcn/ui**: از کامپوننت‌های موجود استفاده کنید
❌ **نوشتن test code**: در این پروژه test code ننویسید
❌ **استفاده از list type در Prisma schema**: primitive types فقط

### ۵.۳. محدودیت‌های مستندات
❌ **ویرایش roadmap.md**: فقط زمانی که فیچر جدید اضافه می‌شود
❌ **ویرایش roadmap-progress.md**: فقط پس از تکمیل فیچر و تأیید کاربر
❌ **ایجاد README یا مستندات اضافی**: مگر درخواست صریح

### ۵.۴. استراتژی GitHub (Dev vs Main) ⚠️ **حیاتی برای امنیت کدها**

**🎯 اصل مهم:**
ایجنت در سرور لوکال آزاد است، اما در GitHub قوانین سخت‌گیرانه دارد.

---

#### 🔧 در سرور لوکال:
- ✅ همیشه روی **branch main** کار کنید
- ✅ آزادید هر کاری بخواهید انجام دهید
- ✅ نیازی به branch اضافی ندارید
- ✅ همه تغییرات در لوکال امن هستند

---

#### 🌐 در GitHub:
- ✅ **همیشه به branch dev push کنید**
- ❌ **هرگز به branch main push نکنید** مگر با اجازه صریح کاربر
- ✅ **Merge از dev به main فقط با اجازه کاربر**

---

#### 📋 گردش کار استاندارد:

```
1. در سرور لوکال روی main کار کنید
   ↓
2. فیچر را بسازید و تست کنید
   ↓
3. به GitHub → به branch dev push کنید
   ↓
4. کاربر تست می‌کند
   ↓
5. کاربر تأیید می‌کند
   ↓
6. درخواست اجازه برای merge به main
   ↓
7. Merge dev → main (فقط با اجازه کاربر)
```

---

#### ⚙️ دستورات Git:

**برای push به GitHub (همیشه به dev):**
```bash
# commit در لوکال
git add .
git commit -m "feat: توضیح فیچر"

# push به GitHub → branch dev
git push origin main:dev
```

**برای merge از dev به main (فقط با اجازه کاربر):**
```bash
# 1. sync با origin/main
git pull origin main

# 2. merge از origin/dev
git merge origin/dev

# 3. حل conflicts (اگر وجود دارد)

# 4. push به origin/main (فقط با اجازه کاربر!)
git push origin main
```

---

#### 🚫 قوانین ممنوعه در GitHub:

❌ هرگز `git push origin main` بدون اجازه کاربر
❌ هرگز بدون اجازه کاربر از dev به main merge کنید
❌ هرگز بدون اجازه کاربر چیزی را مستقیماً به main در GitHub push کنید

#### ✅ قوانین مجاز در GitHub:

✅ همیشه `git push origin main:dev` (push به branch dev)
✅ همیشه قبل از هر کار، از کاربر اجازه بگیرید
✅ همیشه قبل از merge، مطمئن شوید فیچر تست شده است

---

#### 🔑 خلاصه ساده:

| مکان | Branch | قوانین |
|------|--------|---------|
| **سرور لوکال** | main | آزادید هر کاری بخواهید |
| **GitHub** | dev | همیشه به این push کنید |
| **GitHub** | main | فقط با اجازه کاربر |

### ۶.۳. مدیریت Cache و Restart سرور ⚠️ **حیاتی**

**ماهیت مشکل:**
در محیط توسعه Next.js، تغییرات در Prisma Schema یا ساختار دیتابیس در cache فایل‌های `.next` ذخیره می‌شوند. این باعث می‌شود که کدهای جدید (مثل اضافه شدن مدل جدید در Prisma) در runtime اعمال نشوند و ما در چرخه تکراری اصلاح و شکست گیر کنیم.

**نمونه‌های پرتکرار:**
- اضافه شدن مدل جدید در `prisma/schema.prisma` (مثلاً `sharedReport`)
- تغییرات در فیلدهای مدل‌های Prisma
- اجرای `bun run db:push` یا `bun run db:migrate`
- تغییر در تنظیمات دیتابیس

**علائم مشکل cache:**
- `db.newModel` وجود ندارد (`undefined`)
- ارور: `Cannot read properties of undefined (reading 'create')`
- تغییرات در schema اعمال نشده‌اند
- کد کامپایل شده اما با نسخه قدیمی اجرا می‌شود

**رویه استاندارد در مواجهه با این موارد:**
```
۱. تشخیص: بررسی اینکه آیا مشکل مربوط به Prisma/DATABASE/Cache است
۲. توقف سرور: `pkill -f "next dev"` و `pkill -f "next-server"`
۳. پاک‌سازی cache: `rm -rf .next`
۴. ری‌استارت سرور: `nohup bun run dev > /home/z/my-project/dev.log 2>&1 &`
۵. تأیید: چک کردن اینکه سرور به درستی start شده
۶. تست: اجرای مجدد عملیات مشکوک
```

**دستورات آماده:**
```bash
# راه‌حل کامل در یک خط
pkill -f "next dev" && pkill -f "next-server" && rm -rf .next && nohup bun run dev > /home/z/my-project/dev.log 2>&1 &

# چک کردن وضعیت سرور
ps aux | grep -E "(next|node.*3000)" | grep -v grep

# چک کردن لاگ‌ها
tail -30 /home/z/my-project/dev.log
```

**نکات مهم:**
- ✅ همیشه قبل از هر تغییر در Prisma Schema، مطمئن شوید که تغییر قبلی اعمال شده
- ✅ پس از `bun run db:push`، حتماً cache را پاک کنید
- ✅ در صورت تکرار نشدن مشکل پس از restart، حتماً در گزارش ذکر کنید
- ❌ هرگز در صورت مشکوک بودن به cache، بیش از ۲ بار اصلاح نکنید (ابتدا cache را پاک کنید)
- ❌ هرگز از تغییرات در کد برای حل مشکل cache استفاده نکنید (مشکل کد نیست)


### ۶.۴. تست تعاملی (Interactive Testing) 🔄

برای ارورهایی که فقط هنگام تعامل کاربر رخ می‌دهند.

---

### 📋 روال ساده:

1. **ایجنت:** "کد آماده‌ست، برو تست بگیر"
2. **کاربر:** تست را می‌گیرد
3. **کاربر:** می‌گوید **"تست رو گرفتم"**
4. **ایجنت:** لاگ را چک می‌کند و نتیجه را می‌گوید

---

### مثال:
```
ایجنت: "کد PDF آماده‌ست، برو تست بگیر"

[کاربر تست می‌گیرد]

کاربر: "تست رو گرفتم"

ایجنت: "لاگ رو چک کردم. همه چیز اوکیه!"
```

ایجنت (بدون پرامپت از کاربر):
"لاگ رو چک کردم:
✅ درخواست API دریافت شد
✅ کاربر احراز هویت شد
✅ HTML تولید شد
❌ ارور: puppeteer not found

الان این مشکل رو حل می‌کنم..."
```

---

### ⚠️ قوانین:

**برای کاربر:**
- ✅ فقط در ابتدا بگویید "آماده‌ام"
- ✅ بعد از تست، **منتظر بمانید** - پرامپت ندهید
- ❌ نگویید "خب لاگ رو چک کن"
- ❌ نگویید "من تست رو گرفتم، چی شد؟"

**برای ایجنت:**
- ✅ بلافاصله بعد از تست، لاگ را چک کنید
- ✅ بدون منتظر ماندن برای پرامپت کاربر، نتیجه را بگویید
- ✅ ارورها را توضیح دهید و راه‌حل ارائه کنید


## ۷. مدیریت پروژه

### ۷.۱. استفاده از TodoWrite/TodoRead
- ایجاد todo list با جزئیات و priority
- به‌روزرسانی status به in_progress هنگام شروع
- به‌روزرسانی به completed پس از اتمام
- فقط یک todo می‌تواند در هر لحظه in_progress باشد

### ۷.۲. کار با Subagents
- هر subagent باید Task ID داشته باشد (مثلاً: 1، 2-a، 2-b، 3)
- subagent باید قبل از شروع `/home/z/my-project/worklog.md` را بخواند
- subagent باید پس از اتمام به `/home/z/my-project/worklog.md` اضافه کند
- هر بخش جدید با `---` شروع شود
- قالب:
```markdown
---
Task ID: <task id>
Agent: <agent name>
Task: <the task>

Work Log:
- <step 1>
- <step 2>

Stage Summary:
- <results>
```

### ۷.۳. مدیریت وابستگی‌ها
- نصب پکیج‌های جدید با bun در صورت نیاز
- بررسی package.json برای پکیج‌های موجود
- استفاده از recharts برای نمودارها (قبلاً نصب شده)

## ۱۰. اصول طراحی و معماری

### ۱۰.۱. Separation of Concerns
- فرانت‌اند و بک‌اند جداگانه
- کامپوننت‌ها قابل استفاده مجدد
- منطق سمت سرور در API routes
- UI logic در components

### ۱۰.۲. DRY (Don't Repeat Yourself)
- استفاده از کامپوننت‌های مشترک
- abstraction برای logic تکراری
- utility functions در `src/lib/`

### ۱۰.۳. KISS (Keep It Simple, Stupid)
- پیاده‌سازی ساده و واضح
- پیچیدگی غیرضروری ممنوع
- خوانایی کد در اولویت
